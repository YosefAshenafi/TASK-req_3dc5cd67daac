# SmartPark Media Operations — System Design

## 1. Overview

SmartPark Media Operations is an on-prem, offline-first platform that combines two concerns in a single deployment:

1. **Media experience** — a consumer-grade library of audio announcements and short clips that Regular Users browse, favorite, play, and organize into shareable playlists on kiosk tablets and front-desk desktops.
2. **Parking device operations** — ingestion, deduplication, and audit of events from gates, cameras, and geomagnetic sensors, with resilience to intermittent LAN links and explicit replay tooling for Field Technicians.

The system runs entirely on a parking-site LAN with no internet egress, while exposing clean extension points (scanning, telemetry, cloud sync) for a future networked deployment.

**Stack (per `metadata.json`):**
- Frontend: Vue.js 3 + TypeScript, HTML, CSS.
- Backend: Laravel (PHP) exposing REST APIs.
- Data: MySQL 8 (primary), Redis (cache/queues), local disk (media assets + thumbnails).

## 3. Component Responsibilities

### 3.1 Vue.js + TypeScript SPA (Frontend)
- Role-aware routing: `/search`, `/library`, `/favorites`, `/playlists`, `/now-playing`, `/admin/*`, `/devices/*`.
- State: Pinia stores (`auth`, `player`, `library`, `playlists`, `notifications`) with strongly-typed state and actions.
- All DTOs defined as TypeScript interfaces in `src/types/` (e.g., `Asset`, `Playlist`, `DeviceEvent`, `MonitoringStatus`) and shared by the API client.
- UI toolkit: Vuetify or Tailwind + Headless UI for kiosk-first touch targets (min 44 px tap area).
- Search: debounced input (300 ms), filter chips for tag/duration/recency, server-side ranked results.
- "Now Playing" panel persists recent plays in `localStorage` for quick restore on reload.

### 3.2 Laravel API (Backend)
- Authentication: Laravel Sanctum with personal-access tokens stored in HTTP-only cookies.
- Authorization: Policy-based (`UserPolicy`, `PlaylistPolicy`, `DeviceEventPolicy`) + route-level role middleware.
- Domain modules:
  - **Identity**: users, roles, freezes, blacklist, soft-delete/purge.
  - **Media**: assets, tags, thumbnails, playlists, favorites, play history.
  - **Discovery**: search index tables, recommendation candidates, feature flags.
  - **Devices**: events, counters, replays, audit log.
  - **Ops**: monitoring metrics, queue health, ingestion health.
- Queues (Redis driver): indexing, thumbnail generation, recommendation candidate refresh, metrics rollups, purge jobs.

### 3.3 MySQL
- InnoDB, `utf8mb4`, per-site instance.
- Nightly backups to `/var/backups/smartpark`, compressed and retained 14 days.

### 3.4 Redis
- Queue driver for Laravel Horizon.
- Cache store for hot search queries (30 s TTL) and feature-flag lookups.
- Rate-limit store (share-code generation, login attempts).

### 3.5 Local Disk
- `storage/app/media/originals/` — uploaded source files.
- `storage/app/media/thumbs/{id}/{w}.jpg` — 160/480/960 px crops generated by queued jobs.
- `/var/log/smartpark/` — masked application logs.

### 3.6 Device Gateway
- A lightweight service (per device host) that accepts raw device outputs (HTTP POST or file drop in `/var/spool/smartpark/inbox`) and forwards to the Laravel ingestion endpoint.
- Persists outbound events to SQLite with a FIFO capacity of 10,000.
- Retries with exponential backoff capped at 5 minutes; on cap-exceed drops with a warning and increments a `gateway_dropped_events` counter.

## 5. Key Flows

### 5.1 Search + Rank
1. SPA sends `GET /api/search?q=...&tags[]=...&duration_lt=120&recent_days=30&sort=recommended`.
2. Laravel normalizes the query, consults `search_index` with full-text match, applies filters, and joins `recommendation_candidates` when `sort=recommended`.
3. If `feature_flags.recommended_enabled` is `false`, the controller silently rewrites `sort=recommended` → `sort=most_played` and sets `X-Recommendation-Degraded: true` so the UI can show a subtle badge.
4. Results are returned with per-row `reason_tags` when recommended.

### 5.2 Playlist Share + Redeem
1. Owner: `POST /api/playlists/{id}/share` → generates `CODE-8CHAR`, stores in `playlist_shares`, returns code + expiry.
2. Recipient: opens SPA, enters code in the "Redeem" dialog → `POST /api/playlists/redeem {code}` → server validates code, clones playlist rows into recipient's library (new IDs, same items), returns the new playlist ID.
3. Redemption is logged (for audit); the share code remains valid for its TTL unless revoked.

### 5.3 Device Event Ingestion
1. Device → Gateway (HTTP or file drop).
2. Gateway attaches `device_id`, ensures `sequence_no`, generates `idempotency_key` if absent, POSTs to `/api/devices/events`.
3. Laravel:
   - `INSERT ... ON DUPLICATE KEY UPDATE` on `(device_id, idempotency_key)` — duplicates return the original row.
   - Compares `sequence_no` to `devices.last_sequence_no`; if lower, flags `is_out_of_order=true` and queues a reconciliation job.
   - Updates `last_sequence_no` to the new max and `last_seen_at`.
4. Response includes `status: accepted|duplicate|out_of_order|too_old`.

### 5.4 Automatic Degradation
- `MonitoringSampler` (scheduled every 30 s) computes rolling p95 latency and recommendation hit rate.
- On trip, flips `feature_flags.recommended_enabled=false` with a `reason` string and emits `AdminAlertCreated`.
- Admin monitoring page polls `/api/monitoring/status` (10 s).

### 5.5 Media Upload Pipeline
1. `POST /api/assets` (multipart) → validator checks size cap, MIME sniffing, magic-byte signature, SHA-256 fingerprint.
2. Asset row created with `status=processing`; queued jobs:
   - `GenerateThumbnails` (160/480/960 px).
   - `IndexAsset` (writes to `search_index`).
   - `RefreshCandidatesForTag` (if new tag seen).
3. On failure, the asset is soft-deleted and the admin UI surfaces the reason.

### 5.6 Account Deletion + Purge
1. Admin issues `DELETE /api/users/{id}` → sets `deleted_at = now()`.
2. Nightly `PurgeSoftDeletedUsers` job hard-deletes users where `deleted_at < now() - 30d`, cascading to playlists, favorites, play history, and encrypted email.
3. Device events referencing the user (if any) are retained but anonymized.

## 6. Security

| Concern                 | Control                                                                 |
|-------------------------|-------------------------------------------------------------------------|
| Password storage        | Argon2id (Laravel default) with per-user salt.                          |
| Sensitive field at rest | AES-256-GCM via `EncryptedField` cast; key at `/etc/smartpark/field-encryption.key` (0600). |
| Log masking             | Monolog processor strips sensitive keys before write.                   |
| Session security        | HTTP-only, `SameSite=Lax` cookies; Sanctum tokens.                      |
| Rate limiting           | Redis-backed: 60/min/user on writes, 5/h/user on share-code generation. |
| Upload safety           | Allowlist + magic-byte sniff + size caps; rejected files never persisted. |
| Brute force             | Lockout after 5 failed logins, 15 min cooldown.                         |
| CSRF                    | Sanctum CSRF for same-origin SPA requests.                              |
| CORS                    | Restricted to the site LAN origin only.                                 |

## 7. Performance & Stability

- **Thumbnailing**: triggered on upload, queued, generates 160/480/960 px widths via Intervention Image; originals on disk, derivatives in `storage/app/media/thumbs/`.
- **Indexing**: `IndexAsset` job refreshes `search_index` on create/update/delete.
- **Caching**: hot search queries cached for 30 s in Redis.
- **Circuit breaker**: see §5.4.
- **Queue workers**: Horizon with 4 processes under `systemd` with auto-restart.
- **SLO**: API p95 < 800 ms, measured in-app by a middleware histogram.

## 8. Observability (local-only)

- `/admin/monitoring` surfaces:
  - API error rate (rolling 5 min).
  - Queue backlog per queue (Redis).
  - Device ingestion rate, last-seen per device, deduped-event count, out-of-order count.
  - Storage health: free disk on the media volume.
  - Current feature-flag state with last-transition timestamps.
- All metrics come from local MySQL/Redis — no third-party telemetry.

## 9. Deployment

- Single Linux host (Ubuntu LTS) per site.
- `systemd` units: `nginx`, `php-fpm`, `mysql`, `redis-server`, `smartpark-horizon`, `smartpark-gateway@*`.
- Laravel deployed via `git pull` + `composer install --no-dev` + `php artisan migrate --force` + `php artisan config:cache`.
- SPA built with `vite build --mode production` (TypeScript strict mode) and served as static assets by nginx.

## 10. Future-Extension Hooks

- `MediaScanPipeline` interface — currently a no-op; later implementations can plug in on-prem AV scanners.
- `TelemetryExporter` interface — currently writes to local disk; can be swapped for remote shipping.
- `CloudSync` module — scaffolded but disabled; future opt-in replication to a central console.
